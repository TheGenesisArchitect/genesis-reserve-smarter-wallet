This adds a full network hop with no value — pure latency tax. `n Why it is wrong: A BFF (Backend-for-Frontend) that only proxies calls provides zero benefit over calling the backend directly. It adds 30–80ms latency per request, doubles your failure surface, and creates a maintenance burden. A real BFF does three things: (1) data aggregation — one dashboard API call that composes balance + yield + compliance in a single response, (2) view-specific shaping — transform backend DTOs into exactly what the UI component needs, (3) auth delegation — exchange Privy JWT for backend API key at the BFF layer so the frontend never holds a backend credential. `n `n RESOLUTION: Replace "transparent passthrough" with 4 BFF aggregate endpoints: GET /api/gr/dashboard (balance + yield + compliance + epoch), GET /api/gr/history (paginated ledger), POST /api/gr/send (quote + reserve + finalize orchestrated), GET /api/gr/yield (snapshot + allocations + epoch). Each BFF endpoint calls 2–4 backend services in parallel and returns one response. Net latency improvement: 40–60% vs sequential passthrough calls. `n `n P0 `n CRITICAL `n Issue 2: Yield Engine Is Listed as a Minor Balance Field `n What the scope says: "YTD Yield accrued" as a line item under the Account Summary Card. The Yield Engine is treated as a balance attribute. `n Why it is wrong: The Yield Engine is the primary differentiator of Genesis Reserve versus every competitor in the partner intelligence report. It is not a field — it is a live intelligence layer running 15-minute harvest cycles, 4 protocol adapters, per-second balance interpolation, and real-time WebSocket events. Hiding it as a balance line item wastes the most powerful feature in the product. Partners choose Genesis Reserve because their clients earn yield; relegating this to "YTD Yield" suggests it isn't wired. `n `n RESOLUTION: YieldEngineDashboard becomes a primary nav view — not a field. It shows: live blended APY with source badge (LIVE/CHAIN/EST), 24-hour harvest sparkline, per-adapter allocation bar with individual APYs, epoch countdown ring, harvest event log with Arbiscan links, USDC peg monitor. This is already built in YieldEngineDashboard.tsx. It needs to be wired into the main app nav, not hidden. `n `n P1 `n HIGH `n Issue 3: MoneyGram Integration Is Marked as Complete — It Is Not `n What the scope says: "Corridor selection (auto-populated), payout method selector, Submit order" — implied as launch-day functionality. `n Reality: The RemittanceOrchestrator._submitMoneyGram() method in remittance.service.ts contains a comment: "Stub: In production, use MoneyGram Partner API." The FX rates return hardcoded values with random noise. The order status is hardcoded to "IN_TRANSIT". This means the entire send flow is simulated, not live. Launching with simulated payouts to real users would be a compliance and regulatory violation . `n `n RESOLUTION: Three options, ranked by readiness: (1) MoneyGram Ramps API (approved partner program, ~4 week integration), (2) Circle CCTP for US→IN corridor only (stablecoin native, fastest), (3) Mock mode with clear "DEMO" label for testing partners only — NEVER shown to end users. Must be resolved before ANY real user can access the send flow. `n `n P1 `n HIGH `n Issue 4: Load Test Target Exceeds Current Infrastructure Capacity `n What the scope says: "Load test: 1,000 concurrent users, <500ms P95 latency." Also: "Scale: 10k users, 100k+ monthly transactions." `n Why this fails: The current PostgreSQL pool max is 20 connections (db.ts line 12). At 1,000 concurrent API requests, each requiring a DB query, the pool saturates immediately. Requests queue behind each other — P95 becomes 5–15 seconds, not 500ms. Additionally, there is no PgBouncer connection pooler , no Redis caching on read-heavy routes (balance, yield snapshot), and no horizontal scaling configuration (one API process). At 100k monthly transactions, the ledger table will reach ~2–3M rows within 6 months — index strategy must be planned. `n `n RESOLUTION: Before load test: (1) Add PgBouncer sidecar to docker-compose (pool_mode=transaction, max_client_conn=500), (2) Wire Redis cache to 4 high-frequency read routes: GET /balance (30s TTL), GET /yield/snapshot (60s TTL), GET /compliance/status (5m TTL), GET /remittance/order/:id (10s TTL), (3) Add pg indexes: ledger_entries(account_id, created_at), fund_reservations(status, expiry_at), treasury_accounts(wallet_address). Neither is acceptable at enterprise scale. `n `n RESOLUTION: Add a partner_feature_flags table to PostgreSQL: (partner_id UUID, feature TEXT, enabled BOOLEAN, config JSONB, updated_at). This supports one-to-many. But the BFF needs to: (1) resolve which accounts belong to the authenticated user, (2) determine which partner each account belongs to, (3) validate that the user's Privy wallet controls the account. The account_id (pta-xxx format) must be stored server-side and validated on every request — not just held in frontend state. The current middleware has no per-account authorization check beyond account_id format validation. `n `n RESOLUTION: Add an account ownership middleware: verify that the authenticated user_id owns the requested account_id before any treasury operation. Query: SELECT 1 FROM treasury_accounts WHERE account_id = $1 AND owner_id = (SELECT user_id FROM users WHERE privy_did = $2). Cache result in Redis (session-scoped). BFF GET /api/gr/accounts returns all accounts for the current user; account switcher reads from this. Active account stored in BFF session, not localStorage. `n `n P3 `n LOW `n Issue 10: Error Rate KPI Baseline Is Missing `n What the scope says: "Error Rate <1% target." No definition of what counts as an error or how it is measured. `n Why it matters: Without a shared definition, this KPI is unmeasurable. Is a compliance screen that returns FAIL an "error"? Is an FX quote that expires before order submission an error? Is a 4xx response from the BFF an error? Different counting methodologies will produce wildly different numbers and lead to disagreements about whether SLA is met. `n `n RESOLUTION: Define error rate as: (5xx responses + timeout events + payment failures) / (total API requests). Explicitly exclude: 4xx client errors (validation failures, auth failures), compliance FAIL results (expected business outcomes), FX quote expiries (user behavior). Instrument Datadog with separate dashboards for: server_error_rate (5xx), client_error_rate (4xx), compliance_block_rate, payment_failure_rate. Each has its own target. `n `n `n `n `n 2.
